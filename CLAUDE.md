# CLAUDE.md: Lala's Kitchen (Match-3 Engine)

This file is the map. Read it before touching code, since it explains not just what to build but why the boundaries exist. Skipping the "why" is how a reusable engine quietly turns into a single-use game.

## What This Project Actually Is

A reusable match-3 engine, first deployed as a cooking-themed skin called Lala's Kitchen, built for a real audience of one (my mom, who plays Candy Crush to keep her mind occupied and plays with sound off). The engine is the long-term asset. This skin is the first proof it works.

Think of it like a kitchen. The engine is the stove, the pots, and the knife skills, they don't change no matter what's cooking. The skin is tonight's recipe. If you ever catch yourself writing knife-skills logic that only works for tonight's specific recipe, stop, that's the signal something leaked into the wrong layer.

## The Leak Test (the one rule that matters most)

Before writing any code inside `engine/`, ask: if the skin were swapped for something totally different, would this code break or look weird?

If yes, it belongs in `skins/`, not `engine/`. The engine should never contain a string like `"tomato"` anywhere. It only knows piece type IDs, grid coordinates, and behaviors.

## Architecture

```
/engine

  generator.ts      seeded random level generator, lives next to matrix.ts for the same purity guarantee
  gameState.ts      lives, moves, save/load, the paused_awaiting_input state machine
/skins
  /lalas-kitchen
    config.json     pieceTypes, blockers, lives, animationProfile, palette, recipeCards
    /sprites
/components
  Board.tsx         reads engine output + active skin config, renders, never hardcodes piece names
/services
  adService.ts      AdService interface + selectAdService(platform) — the ad/monetization seam, platform-selected, no real SDK wired in yet
  soundService.ts   SoundService interface + selectSoundService(realService) — resolves to the real expo-audio-backed adapter (services/expoAudioSoundService.ts), no platform branch needed
  hapticsService.ts HapticsService interface + selectHapticsService(platform, native, web) — real expo-haptics on native, no-op on web
```

## Build Order

Build and fully test each phase before starting the next one. Do not jump ahead, the engine's value depends on Phase 1 being solid before anything gets built on top of it.

1. **Core matrix** (`engine/matrix.ts`), pure functions, no UI, no React
2. **Seeded generator** (`engine/generator.ts`), deterministic, same seed always produces the same board
3. **Game state and persistence** (`engine/gameState.ts`), lives, moves, save/load, event emitter
4. **Skin config** (`skins/lalas-kitchen/config.json`), pure data, no logic
5. **Presentation layer** (`components/`), React Native + Reanimated, reads engine output only

Full phase details, including the actual config schema and per-phase build notes, live in `lalas-kitchen-build-spec.md` in this repo. That file is the recipe card. This file is the kitchen rules.

## Testing Philosophy (the fun part, genuinely)

Everything in `engine/` is a pure function: same board in, same board out, no side effects. That means bugs get debugged with a test file, not a phone in your hand.

If a cascade doesn't resolve the way you expect, don't reach for the simulator first. Paste the exact board state into a test, call the function directly, and read what comes back. This is strictly more fun than tapping through fifteen screens hoping to reproduce a bug, and it's a direct rep for the same telemetry-over-assumption habit that caught the RLS bug on GoldStar.

Minimum test coverage before a phase counts as done:
- A board with zero matches
- A board with exactly one obvious three-in-a-row
- A board that cascades at least twice
- A board with zero legal moves (should trigger `shuffle`)

## Data Model Notes

Every piece object carries a `type` field from day one, so a future special piece becomes a config addition instead of a schema migration. `'blocker'` is no longer just the placeholder this note originally described — blockers are real as of the adjacent-damage clearing mechanic (see `engine/DECISIONS.md`'s Phase 6 section): not matchable, not swappable, cleared by taking one hit per adjacent match until `hitsRemaining` reaches zero, at which point the clear counts toward an objective exactly like any piece type's `matchType`. `'striped'` is the first special piece, now built (it replaced the old `'row_clearer'` placeholder — see `engine/DECISIONS.md`'s striped-piece entry): a run of exactly 4 converts one cell into a striped piece carrying the base `matchType` plus a `direction` (`'row'`/`'col'`), and matching that striped piece later — it's an ordinary matchable, swappable piece until then — sweeps its whole row or column. A horizontal 4-run makes a row-clearer, a vertical one a column-clearer (parallel mapping; the perpendicular genre convention is a one-line flip, see the DECISIONS entry). It renders through the same sprite path as every other piece: `components/spriteMap.ts`'s `getSpriteForPiece` resolves a striped piece to `striped_<its base sprite>`, so dedicated art is one `spriteRegistry.ts` line per asset and any type without it falls back to the standard text-label placeholder (tomato and lemon have real striped art; the other four fall back for now).

`'color_bomb'` is the second special piece, now built (see `engine/DECISIONS.md`'s color-bomb entry): a run of *exactly 5* (row or column) converts one cell into a colorless `type: 'color_bomb'` piece (no `matchType` — it can't form an ordinary run, excluded by `piecesMatch` like a blocker; a 5-run therefore credits 4 toward objectives, the anchor becoming the bomb). Its activation is a genuinely different mechanism from the striped piece's, not an extension of it: a striped piece fires by *being included in a later match*, but a color bomb fires the instant it's *swapped with any piece* — regardless of whether that swap would form a match — and clears every piece on the board sharing the swapped-with piece's `matchType` (swapping two bombs clears the whole board). This lives in `gameState.ts`'s `applyMove` as a branch that runs **before** the ordinary no-match snap-back check (a bomb swap is always a legal, committed move precisely because it doesn't rely on a run), with `resolveColorBomb` doing the detonation + refill and handing off to the normal `resolveCascades` for any chain matches; `hasLegalMoves` also treats any bomb-involving swap as legal so a board whose only move is a bomb swap is never wrongly judged stuck. Blockers are **never** force-cleared by a detonation (single-type or whole-board) — they only ever take normal one-hit-per-call adjacent damage, the same rule as every other clearing mechanism. It renders through the same `getSpriteForPiece` path (a fixed `'color_bomb'` sprite key — note: no `.webp` extension, so the `spriteRegistry.ts` entry is keyed by the bare `'color_bomb'` string, unlike every other filename-keyed entry). Real dedicated art (`color_bomb.webp`, a glowing potion bottle) has landed as one registry line; before it, a bomb fell back to the "CO" text-label placeholder. Both states verified live, see `docs/verification/color-bomb/`. Its detonation also now has its own presentation-only identity — a radial ripple (`components/specialEffectAnimation.ts`'s `radialDelaysForClears`) expanding outward from the swapped position, normalized to one fixed calm duration regardless of board size, replacing the flat instant vanish every clear used to share (see `engine/DECISIONS.md`'s special-effect-identity entry and `docs/verification/special-effect-identity/`). Only the bomb's own swap-triggered detonation gets this; a color bomb reached via chaining still clears with the plain generic treatment (see `DEFERRED_COMPLEXITY.md`).

Special-piece **combos** are now built (see `engine/DECISIONS.md`'s special-piece combos entry): swapping two special pieces directly into each other fires a combined effect on the swap itself, bypassing the snap-back exactly like a solo bomb. Two exist — **striped + striped** clears a full cross (the entire row *and* column through the swap, overriding each piece's own direction), and **striped + color bomb** is a super-combo that converts every board piece of the striped piece's `matchType` into a striped piece and fires them all at once (`resolveStripedCross` / `resolveStripedBombCombo`, both routed through the shared `resolveClearSet` tail). Branch order in `applyMove` is load-bearing: striped+bomb is checked **before** the solo-bomb branch, or `resolveColorBomb` would consume it and run the weaker single-type clear; striped+striped is checked before the ordinary snap-back, since two stripeds needn't form a run. `hasLegalMoves` treats a striped+striped pair as always legal (striped+bomb already is, via the bomb). Blockers caught in either combo take normal one-hit adjacent damage, never a force-clear. Verified live, see `docs/verification/special-piece-combos/`. Still deferred: L/T-shape triggers (an L/T-formed bomb, vs. the built straight-5 trigger) — see `DEFERRED_COMPLEXITY.md`. (Sweep/detonation **chaining** — a special caught in a combo's clear firing its own effect — is now built; see the chaining paragraph below.) Both combos now also have their own presentation identity, not the flat generic vanish (see `engine/DECISIONS.md`'s special-effect-identity entry): the cross combo reuses the striped piece's own existing traveling-sweep visual unchanged, only correcting the delay computation to a true bidirectional center rather than each swapped piece's own (previously mismatched) individual direction; the supercombo plays two real beats — a brief "converting" flicker on every converted piece, then a synchronized pop-and-shrink together — which is what the convert-to-striped animation flash noted above as deferred now resolves to (the engine still computes the settled clear in one step; only the animation shows two beats). Verified live, see `docs/verification/special-effect-identity/`.

`'area_bomb'` is the third special piece, now built — and then **reversed from passive/colored to active/colorless** after real play (see `engine/DECISIONS.md`'s area-bomb entry and its reversal sub-entry). **Spawn:** a **2×2 square** of four same-`matchType` pieces converts its top-left cell into a `type: 'area_bomb'` piece and clears the other three (so a 2×2 credits 3 toward objectives, the anchor becoming the bomb — the same accounting a 4-run→striped uses) — **unless one of the four cells is already a live striped piece**, a real-play gap fixed after the fact (see `engine/DECISIONS.md`'s "Squares: a live striped corner now counts" entry): that pre-existing special now fires its own line sweep instead of a new area bomb spawning, mirroring how a run containing a live striped piece fires it rather than spawning a second special. `matrix.ts`'s `checkSquares` corner gate was relaxed from strictly `type === 'normal'` to also allow a live `'striped'` corner (blocker/void remain excluded; color/area bombs are moot, being colorless). Verified live, see `docs/verification/square-striped-corner/`. It's the first special triggered by a shape that isn't a straight line, which forced a genuinely new detector: `matrix.ts`'s `checkSquares` scans for 2×2 blocks alongside — never folded into — the run-based `checkMatches` (a pure 2×2 has no 3-in-a-row, so the run scan alone misses it entirely). Because a square is a new match *shape*, every "is there a match?" gate consults **both** scans — `applyMove`'s snap-back, `resolveCascades`'s loop, `hasLegalMoves`, and the generator/`shuffle` match-free guarantees — or a square-forming swap would snap back as a no-match move and a latent square from generation would spawn a *free* bomb on the player's next move. **Activation is now active and colorless, the same camp as the color bomb** (this reverses the original passive/colored fork): the spawned bomb **drops its `matchType`** — `piecesMatch` excludes it exactly like a color bomb, so it can never form or join a run — and it fires the instant it's **swapped with any ordinary piece**, clearing the **3×3** block centered on itself (`areaBlastPositions` → `resolveAreaBomb` → the shared `resolveClearSet`), regardless of the partner's type or whether a run would form. The reversal was chosen because the single universal `area_bomb.webp` sprite (below) showed no color, so a player could never tell what match a *passive* bomb needed — going colorless makes that question moot. This required the same shortcuts the passive design had avoided: a **new `applyMove` swap branch** (placed **before** the solo-color-bomb branch — an `area + color_bomb` swap is "bomb-involving," so `resolveColorBomb` would otherwise run a degenerate clear on the area bomb's `undefined` matchType) and a **new `hasLegalMoves` clause**. Swapping an area bomb into **another special** (color bomb, striped, or another area bomb) now has a real combined effect for all three pairings — the original deferred-combo snap-back is gone (see `engine/DECISIONS.md`'s "Area-bomb combos: the last three pairings" entry): **area + color bomb** converts every piece of the board's most-common matchType (`mostCommonMatchType`, reused from the chained-partner-less-bomb case) into an area bomb and fires them all at once, several 3×3 blasts simultaneously; **area + striped** fires a plus shape, the area bomb's 3×3 block unioned with the striped piece's own full sweep line, built from the existing `areaBlastPositions`/`sweepLinePositions` helpers with no new geometry; **area + area** fires one bigger 5×5 blast rather than two separate 3×3s (`areaBlastPositions` gained an optional `radius` parameter, default 1, so every existing call site is unaffected). All three route through the same shared `resolveClearSet` every other special effect uses, and `hasLegalMoves` now treats every area-bomb pairing as legal. Verified live, see `docs/verification/area-bomb-combos/`. A colorless bomb's own cell now credits nothing on detonation (like the color bomb); the cleared neighbours still credit by their matchType. A 2×2 whose cells overlap a straight run **usually stands down** — the run logic owns those cells — but a real playtest report surfaced a gap: when the square is the ONE unambiguous embedding (every overlapping run is exactly length 3, no other detected square shares a cell with it, and it shares no cell with a detected cross), it now fires anyway instead of standing down (see `engine/DECISIONS.md`'s "Embedded square in a straight run" entry). A genuinely ambiguous region — two or more overlapping square candidates, e.g. a full aligned 2×3 rectangle where both rows are independently exactly-3 runs — still stands down, explicitly, not silently guessed at. Blockers caught in a 3×3 blast take normal one-hit adjacent damage, never a force-clear. It renders through the same `getSpriteForPiece` path as a **single fixed** `area_bomb.webp` (one sprite for every area bomb, like the color bomb — but keyed by the real filename, not extensionless); because the piece is now colorless there was never a per-type sprite to derive, so the reversal needed **zero** rendering changes. Verified live, see `docs/verification/area-bomb/active/`. `area + special` combos (an area bomb swapped *directly into* another special — distinct from chaining, which is a special *caught in a clear*) are now built too, described above. Still deferred: a genuinely *ambiguous* square-overlapping-a-run (two or more overlapping square candidates — the now-fixed *unambiguous* case is described above), and any L/T intersection involving a 4- or 5-long arm (vs. the now-built exact-3×3 crossing) — see `DEFERRED_COMPLEXITY.md`. (Blast **chaining** — an area bomb caught in a 3×3 blast firing its own 3×3 — is now built; see the chaining paragraph below.) The area bomb also carries **two presentation-only powder animations** (see `engine/DECISIONS.md`'s "two powder animations" entry): a calm idle wisp drifting up from the bag while it rests (reusing `SteamWisp`'s rise/fade loop, gated by a `powderWisp` prop `Board.tsx` sets from `piece.type === 'area_bomb'`, the same shape as `direction`/`spreadWarning`) and a trigger poof puffing outward from the bag as the 3×3 clears (the `isPowderBurst` branch of `Tile.tsx`'s `ExitingTile`, on the clear's own `matchDurationMs` clock, rendered in a **sibling** view so the cloud grows while the bag shrinks). The engine is untouched — no new `Piece` field. Verified live, see `docs/verification/area-bomb/powder/`.

A **second, additive area-bomb trigger** was later built: `matrix.ts`'s `checkCrossShapes` finds every crossing-run point — a cell where a horizontal run and a vertical run, each **exactly length 3**, meet (the classic L/T/plus) — and spawns an area bomb there too, credited the same 4-objectives-plus-1-anchor way a 5-run's color bomb is (a pure 3×3 cross is always exactly 5 cells: 3+3−1 shared corner). Unlike the 2×2 square, a crossing's own two arms are each already an ordinary `checkMatches` run, so **no** `hasLegalMoves`/`shuffle`/`applyMove`/generator legality-gate wiring was needed (only `checkSquares` needed that, since a pure 2×2 forms no run at all) — verified directly against the code, not assumed. A crossing point involving a 4- or 5-long arm, or a live striped piece anywhere in its five cells, stands down entirely — the existing run logic (or that striped piece's own sweep) fires unchanged, unaffected by this addition. See `engine/DECISIONS.md`'s crossing-run entry and `docs/verification/crossing-shape/`.

**Special-piece chaining** is now built (see `engine/DECISIONS.md`'s special-piece-chaining entry) — the deferred behavior, carried since the striped piece first shipped, where a special caught in another special's clear should fire its own effect too, not just vanish as ordinary content. It lives in one pure `gameState.ts` helper, `expandChainClears`: given the cells an effect is about to clear, it folds in the clear set of every OTHER special the seed catches — a striped piece sweeps its line, an area bomb blasts its 3×3, a color bomb (which has no swap partner in a chain) detonates the board's **most common `matchType`** (`mostCommonMatchType`, deterministic) — repeating to fixpoint. The investigation confirmed `resolveClearSet` is the single shared funnel every swap-triggered effect already passes through (solo color bomb, area bomb, both combos), so chaining is wired in there **once**; it's also wired into `resolveMatchEffects`'s in-match striped sweep (the one clear path that doesn't route through `resolveClearSet` — a striped piece included in an ordinary run), since swapping a striped into a match is the commonest way a player fires one and leaving that path un-chained would read as inconsistent (architect-confirmed both sites). Each caller passes `originKeys` — the specials **it** already fired/consumed (its own bomb, its swapped stripeds, the supercombo's converted pieces, the in-match striped triggers) — which still clear but never re-fire, so a chain is only ever the specials an effect *catches*, never the ones it *is*. A chain is a **free bonus** on the original single move — it never costs an extra move — and each triggered effect still credits its own cleared cells to objectives (the existing per-cell matchType counting runs over the fully-expanded set). It **always terminates**: finitely many cells, each cleared at most once, only a fresh non-origin special is ever enqueued. The whole chain folds into one committed clear set (per-link animation staging is a deferred presentation nicety, like the supercombo flash). Verified live (real `applyMove`, real sprite path), see `docs/verification/special-piece-chaining/`. Still deferred: that per-link animation flash — see `DEFERRED_COMPLEXITY.md`.

**First-time special-piece tutorials** are now built (see `engine/DECISIONS.md`'s special-piece tutorial overlay entry). Each special (striped, color bomb, area bomb) shows a calm one-time "here's what this does" card the first time that piece comes to rest on the board — the sibling of the existing blocker tutorial ("A Covered Dish"), reusing `SaveData.seenTutorials` + `markTutorialSeen` unchanged, gated behind input and dismissed with a single "Got it". It's **one data-driven component** (`components/SpecialTutorialOverlay.tsx`), not three files: only the headline/subtext (`SPECIAL_TUTORIAL_CONTENT`, keyed by tutorial id) and the icon differ, and the icon resolves through the same `getSpriteForPiece` path every live tile uses (a striped piece shows its real `striped_<base>` art). The tutorial ids **are** the engine `PieceType` strings (`'striped'`/`'color_bomb'`/`'area_bomb'`), so `appPersistence.ts`'s `findSpecialPieceTutorial` compares `piece.type` straight against the id with no mapping table. Unlike the blocker tutorial's mount-time check, a special never exists on a level's *initial* board (the player forges it mid-level), so `Board.tsx` re-derives it in a post-move effect keyed on `gameState`, showing the FIRST unseen special row-major (a second minted the same move shows after the next move — overlays never stack), with a session-level `dismissedSpecialTutorialsRef` folded into the seen check so a just-dismissed card can't flash back before the persist round-trips. Verified live, see `docs/verification/special-piece-tutorial/` — including a follow-up capture (`organic-spawns/`) confirming all three tutorials fire from a genuine in-game spawn (a real 4-run, 5-run, and 2x2 square, each forged by real tap gestures against the real running app), not just the striped panel as the original capture proved; the color-bomb and area-bomb panels had originally only been verified against a pre-placed resting piece.

A **fourth tutorial, `chain_reaction`**, is now built on top of these three (see `engine/DECISIONS.md`'s `chain_reaction` entry) — the actual differentiator of this whole game: the moment more than one special piece fires *together* from a single move, via chaining (a special caught in another's clear firing its own effect too) or a combo. Its trigger is a pure by-product of the chaining work's own `originKeys`/`expandChainClears` bookkeeping, not new detection: `engine/gameState.ts` counts, per pass, how many already-special pieces fired their own effect into that pass's clear set (`countFiredSpecials`, threaded through as `CascadeResolution.maxSpecialsFired`), and exposes `ApplyMoveResult.multiSpecialFired` once that reaches 2 within a single pass (a max across cascade passes, not a sum — the teachable moment is specials compounding in the SAME resolution step). Because the specials that fired are, by definition, already cleared by the time the move settles, this can't be a board-scan tutorial like the other three: `appPersistence.ts`'s `shouldShowChainReactionTutorial(multiSpecialFired, seenTutorials)` is a plain once-ever gate over that boolean instead. It reuses the exact same `SpecialTutorialOverlay` component, whose `piece` prop is now `Piece | null` — chain_reaction has no single piece to anchor its icon to (it celebrates a moment, not a piece), so it falls back to the same `spriteLabel` placeholder convention every un-arted sprite already uses. `Board.tsx`'s `animateCascade` sets it at the same point the combo-streak banner fires, which — via React's batching and the existing per-piece effect's own "don't stack" guard — naturally gives chain_reaction priority over an unrelated freshly-spawned special on the same move with no new priority logic. Verified live with the genuine-organic-spawn requirement built in from the start (not a follow-up closing the gap the way the three-tutorial capture needed to): a hand-built, match-free board where a real two-tap swap makes a color bomb detonation genuinely chain into a caught live striped piece, sweeping a whole column — see `docs/verification/chain-reaction-tutorial/`.

A genuine **first-time onboarding tutorial, `how_to_play`**, is now built (see `engine/DECISIONS.md`'s "How-to-play onboarding tutorial" entry) — the one card that comes *before* all the tutorials above, since every one of those assumes the player already knows how to swap tiles. It's a calm, one-time "here's the base mechanic" card (tap or drag a tile toward a neighbor to swap, line up three or more to clear), shown the very first time a genuinely fresh save's level 1 loads. Confirmed nothing like it already existed (a repo-wide search for `howToPlay`/`onboarding`/`welcome` turned up nothing beyond an unrelated Home welcome-back string) before building it. It reuses `SpecialTutorialOverlay` unchanged rather than a sixth overlay file — a sixth `SPECIAL_TUTORIAL_CONTENT` entry (`how_to_play`, `appPersistence.ts`'s `HOW_TO_PLAY_TUTORIAL_ID`) with the same `piece: null` fallback `chain_reaction` already established (no single piece to anchor an icon to, since it explains the mechanic itself before any piece has been swapped). The gate is deliberately **not** just `levelIndex === 1`: `appPersistence.ts`'s `shouldShowOnboardingTutorial(levelIndex, completedLevels, seenTutorials)` also requires `completedLevels.length === 0`, since a player who already finished level 1 (or further) and later replays it from All Levels or "Play again" would also have `levelIndex === 1` despite already knowing how to play — `completedLevels` is the actual "has this account ever won anything" signal, and is never empty again once it isn't. `Board.tsx` takes a new `completedLevels` prop (App.tsx already tracked this state) and decides at mount, the same shape `showBlockerTutorial` already uses, and it takes top priority over every other tutorial (gates `canAcceptMove`/`dragEnabled` alongside the existing checks, so no move — and therefore no special piece or blocker interaction — is even possible until it's dismissed). Verified live against the real running app over CDP, including the once-ever guarantee across a genuine page relaunch (not just a single mount): `localStorage.clear()` for a genuinely fresh save, a real click on "Start cooking," the overlay rendering over the real level 1 board, then dismiss → persisted `seenTutorials` updates immediately → a full reload and re-entry into level 1 shows no overlay at all. See `docs/verification/how-to-play-tutorial/`.

**Dynamic denial-zone spread** is now built (see `engine/DECISIONS.md`'s Phase 8 entry). It builds *on* the blocker system, never replacing it: the investigation confirmed that a calm **static** denial zone — several cells blocked off, clearable only by ordinary or special matches landing on them — already needs **zero new engine logic**, because that's exactly the existing blocker contract (`piecesMatch`/`applyMove` exclusions + adjacent-damage clearing); clustering blockers into a region is a level-content placement choice, not a mechanic. So **every level below the difficulty threshold uses static blockers unchanged**. The *dynamic* layer, gated to generated levels at or past `DENIAL_SPREAD_MIN_LEVEL_NUMBER` (10) in `appPersistence.ts` — the same `generatedLevelNumber` gate shape `pot_lid` uses, deliberately later than pot_lid's 7 — makes an unaddressed zone **grow into an adjacent ordinary cell**. Its runtime state is `GameState.denialSpread` (`DenialSpreadState`), present only on gated levels (`undefined` below the threshold, so `applyMove` skips the spread branch entirely — static blockers, unchanged). The spread cadence is **proportional to the level's own move budget**, not a universal number: `spreadInterval = max(2, round(movesLimit × SPREAD_MOVE_FRACTION))` with `SPREAD_MOVE_FRACTION = 0.25` (an 18-move level spreads every 5 unaddressed moves, a 30-move level every 8, a real level-10/20-move board every 5), so the pressure reads the same regardless of level length; the `max(2, …)` floor guarantees a warning move always precedes a spread. Each committed move, `applyMove` derives whether the zone was **addressed** by comparing total blocker `hitsRemaining` before vs. after the cascade settles (a strict decrease = any blocker damaged/cleared) — addressed resets the spread clock and cancels the warning, which is why matching the warned cell (adjacent to the blocker by construction) defuses the growth. `matrix.ts`'s `findSpreadTarget` (a pure scan, sibling to `hasLegalMoves`) picks the deterministic frontier: the first blocker in row-major order bordering an `'normal'` cell, paired with its first ordinary neighbor in `ADJACENT_OFFSETS` order — only ordinary cells are eligible, so a spread never eats a player-earned special or double-places onto a blocker. `gameState.ts`'s `stepDenialZone` (run in `applyMove` **before** the legal-move rescue, so a spread-created blocker is covered by the same `hasLegalMoves → shuffle` guarantee) either resets (addressed), marks the frontier cell with a new optional `spreadWarning: boolean` `Piece` field one move before the spread (`interval - 1`), or converts that cell into a blocker (inheriting the source blocker's `matchType` and the level's `blockerHitsToClear`, **reusing the cell's id** so it morphs in place with no clear/spawn diff) on the spread move (`interval`). `spreadWarning` is presentation-only — invisible to `piecesMatch`, so the warned cell stays an ordinary matchable/swappable piece. It renders as a calm **crack + dimming glow** via `components/Tile.tsx`'s `SpreadWarningOverlay` (steady dark wash + accent crack + slow ~900ms breath, never a flashing alarm — per the calm brief), wired through `Board.tsx`'s `spreadWarning` prop. Verified live, see `docs/verification/denial-zone-spread/`. Still deferred (see `DEFERRED_COMPLEXITY.md`): spread eats only ordinary cells (never specials); no spread chaining/merging beyond becoming one more ordinary blocker.

**Generator-driven clustered denial-zone placement** is now built (see `engine/DECISIONS.md`'s clustered-denial-zone-placement entry): a denial-zone-eligible generated level (`generatedLevelNumber >= DENIAL_SPREAD_MIN_LEVEL_NUMBER`) now places its blockers as **one contiguous region** from the moment the level loads, rather than the original independent `fisherYates` scatter — the zone no longer needs several spread moves to grow visible contiguity in. `engine/generator.ts`'s new `clusteredPositions` (gated by `GeneratorConfig.clusterBlockers`) picks a random start cell, then repeatedly claims a random cell off the frontier of cells adjacent to the region grown so far, reseeding a fresh region if the frontier ever dries up before reaching the requested count (a shape template splitting the board into disconnected pockets). `engine/gameState.ts`'s `createGameState` is the one place the actual gameplay flag (`LevelConfig.denialSpread`) is translated into this placement-only `clusterBlockers` switch — `generator.ts` itself has no concept of a "denial zone," only "grow one region or scatter." Every level below the threshold, and every hand-built level, is byte-identical to before (the original scatter path is untouched). Verified live, see `docs/verification/clustered-denial-zone-placement/`.

**Non-rectangular boards (void cells)** are now built (see `engine/DECISIONS.md`'s board-shape / void-cell entry). The premise that this was "designed in from the beginning" **did not hold** — the only board-shape variety that ever existed was per-level rectangular `rows`/`cols`; there was no cutout/hole mechanism anywhere, so it was built from scratch as a real engine feature. A cell is made non-playable by a **sentinel `type: 'void'` piece** — the same shape `'blocker'` uses, chosen over a nullable `Board` so `Board = Piece[][]` stays non-null and every existing reader keeps a real object at every index (a permanent `null` would collide with `calculateCascades`'s transient "just-cleared, awaiting refill" `null`). A void carries no `matchType`, is excluded by `piecesMatch`/`checkSquares`/`hasLegalMoves`/`applyMove` exactly like the other non-content types, is held **fixed** by `shuffle`, is never filled by the generator (or covered by a blocker), and renders as **nothing** in `components/Board.tsx` (`return null` in the tile map — the board background shows through as the cutout). The one genuinely new algorithm is **gravity**: `matrix.ts`'s `calculateCascades` now walks each column as a series of maximal **non-void segments** separated by voids and compacts/refills each segment independently — a piece rests *on top of* a void instead of falling through it, and a refill enters at the **top of its own segment** (so an enclosed pocket, like a plus arm's middle row, refills from its local top rather than draining forever; a diagonal/flow model was rejected as overkill). A void-free column is one full-height segment, so a plain rectangle behaves exactly as before. The shape is declared per level via the new optional **`LevelConfig.voidCells?: Position[]`** (mirror `GeneratorConfig.voidCells`), passed straight through `createGameState` → `generateLevel`; omitted means a plain rectangle, so every existing level is byte-identical. The first non-rectangular level is the hand-built **Level 4 "Cutting Board"** (App.tsx's `LEVEL_QUEUE`) — a plus on a 7×7 grid (four 2×2 corner blocks voided, 33 playable cells). Verified live, see `docs/verification/board-shape/`. **Void × specials interaction was fixed after real playtesting surfaced a regression** (see `engine/DECISIONS.md`'s "Fixed: the ... claim above didn't hold" entry): a striped sweep, an area-bomb blast, a color-bomb whole-board detonation, and the chain reaction any of those trigger were each written before voids existed and only ever excluded `'blocker'` cells from their clear set, never `'void'` — so on a shaped board one of these could reach straight through a void, which `gameState.ts`'s segmented-gravity refill then read as an ordinary gap and permanently refilled, erasing part of the board's shape (and, on the way out, resolving the swallowed void to the exiting-tile pipeline's `'?'` placeholder — the same bug class as the original color-bomb "?", one layer upstream). Fixed with one shared `isClearable` predicate (`type !== 'blocker' && type !== 'void'`) used at all five clear-set-building call sites, mirroring the `swappable` guard `matrix.ts`'s `hasLegalMoves` already used for the same pairing. Verified live, see `docs/verification/void-specials-clear-fix/`. **Generator-driven shapes are now built too** (see `engine/DECISIONS.md`'s "Generator-driven board shapes" entry): `engine/boardShapes.ts` is a small curated set of 3 reusable templates (`cut_corners`, `plus`, `ring`, each a pure `(rows, cols) => Position[]`), and `appPersistence.ts`'s `generatedShapeId(levelNumber)` gates their introduction the same way `pot_lid`/denial-spread are gated, wired straight into `buildGeneratedLevelConfig`'s `voidCells` output. `generateLevel` itself needed zero changes, since the void-cell contract was already fully general. Verified live, see `docs/verification/generator-driven-board-shapes/`. **This gate was later loosened significantly** (see `engine/DECISIONS.md`'s "Board shapes: gating loosened" entry): the original levelNumber-8-then-1-in-4 cadence was reasoned entirely around a difficulty concern that the playableRatio-scaling fix below already resolved — with that gone, `SHAPE_MIN_LEVEL_NUMBER` dropped to 1 (eligible from the very first generated level) and `SHAPE_CADENCE` dropped to 2 (half of all generated levels, not one in four), since the real goal is now visual distinctiveness within the first few minutes of play, not a rare surprise. A second, guaranteed hand-built shaped level, "Pantry Corners" (`cut_corners` at 8x5, `App.tsx`'s `LEVEL_QUEUE`'s 7th and last entry), was also added so a brand-new player sees two distinct curated shapes from hand-built content alone before ever reaching the generator. **The difficulty ramp's blindness to shape was fixed after a real playtest report** (see `engine/DECISIONS.md`'s "generated-level difficulty ramp was blind to shape-template playable area" entry): a `ring`-shaped level felt genuinely unfair, and investigation confirmed why — `ringVoids` leaves only 22 of 40 cells playable at the real generated-board size (55%, versus `cut_corners`' 70% and `plus`'s 80%), while `buildGeneratedLevelConfig`'s `movesLimit`/`targetCount` were computed purely from `levelNumber`, with zero awareness of how many cells a shape had just removed. `engine/boardShapes.ts`'s new `playableCellRatio(rows, cols, voidCells)` feeds that awareness into `generatedMovesLimit`/`generatedTargetCount` (both gained an optional `playableRatio` param, default `1` — every prior call site is unaffected), scaling each proportionally down before re-applying their existing floors (`MIN_MOVES` 18, and a new `MIN_TARGET` 10) so a heavily-voided board still can't go trivial or unwinnable. Confirmed live on the real first generated `ring` level (generated level number 16): total target dropped from 26 to 14 (a ~46% cut, matching the 55% playable ratio), while `movesLimit` stayed at 18 since it was already pinned to its floor at that level number. Still deferred (see `DEFERRED_COMPLEXITY.md`): void × denial-spread interaction (a spread already only ever targets `'normal'` cells, so it was never exposed to this bug), and the ramp's own step sizes/floors/cap remain hand-picked, not playtested. **The disclosed cut-corners-twice-in-a-row seam at the level 7→8 boundary was later closed after a real playtest report** ("the same board shape keeps appearing") — see `engine/DECISIONS.md`'s "A disclosed, accepted cosmetic overlap" entry's follow-up. Investigation confirmed the rotation itself had no bug (`generatedShapeId`'s gating and template-selection share one input, but `SHAPE_CADENCE`/`BOARD_SHAPE_ROTATION.length` are coprime, so it cycles cleanly); the symptom traced entirely to the already-disclosed seam, amplified by shape being a pure function of level number (so retrying a hard level 8 never varies it) and by early players concentrating play right at that boundary. Fixed with `appPersistence.ts`'s new `SHAPE_ROTATION_OFFSET = 1`, which shifts the generator's rotation start by one step so raw level 8 lands on `plus` instead of `cut_corners` — `plus` chosen over `ring` for the same "gentlest first" reasoning Pantry Corners itself used (80% playable vs. `ring`'s 55%). Pantry Corners' own `cut_corners` choice was left untouched, being reasonable on its own merits — only the generator's independent starting point moved. Verified by a live trace of the real `LEVEL_QUEUE.length` (7) across levels 1–26 confirming zero adjacent shape repeats anywhere, including the 7→8 boundary, and that every template still appears exactly once per 3 shaped levels in clean round-robin order.

`GameState.objective`/`LevelConfig.objective` are no longer a single item — both are now `objectives`, an array of the same per-item shape (`type`, `targetMatchType`, `targetCount`, `currentCount`). A single-objective level (every hand-built `LEVEL_QUEUE` entry today) is just an array of length one, not a special case. Win requires every entry to reach its target (see `engine/gameState.ts`'s `applyMove` and `engine/DECISIONS.md`'s multi-objective entry). Generator-driven levels only ever place a second objective once the level's own piece-type pool has grown to at least 5 distinct types (see `appPersistence.ts`'s `generatedObjectiveCount`) — deliberately late, since a second objective drawn from a small pool makes nearly every random match satisfy some target, and two objectives on one level are always distinct `targetMatchType`s — never the same piece type twice. The piece-type pool itself now grows as generated levels continue (fewer types early, more later — see `engine/DECISIONS.md`'s "Difficulty tuning" entry), the opposite direction from an earlier, incorrect version of this ramp.

**A second objective type, `'score'`, is now built** alongside `'collect'` (see `engine/DECISIONS.md`'s scoring-system entry) — a real scoring system built from scratch, since no concept of score existed anywhere in the engine before this session (confirmed by direct investigation, not assumed). `Objective.targetMatchType` is now optional: undefined for a `'score'` objective, whose `currentCount` tracks cumulative move-score for the level instead of a matchType's clear count. Every cleared cell scores at one of three tiers (`ScoreTier`: `'ordinary'`/`'special'`/`'bomb'`, 10/25/50 points/cell — `engine/gameState.ts`'s `SCORE_TIER_POINTS`): a plain 3-match (or a blocker cleared by adjacent damage) is `'ordinary'`; a 4-run/square-spawned area bomb/striped-piece trigger is `'special'`; a 5-run (or 6+), a solo color-bomb detonation, and either special-piece combo (striped+striped, striped+bomb) are `'bomb'` — combos deliberately share the top tier rather than getting a fourth tier above it (a real judgment call, see the DECISIONS.md entry). Cascades and chains contribute their own points at an increasing per-pass multiplier (`passScoreMultiplier`, +25%/pass) rather than flat per-cell scoring, so a longer chain is worth more than the same cells cleared across separate moves — matching how this game already celebrates chain depth elsewhere (the `combo_streak` event). One hand-built level exercises it: App.tsx's `LEVEL_QUEUE` level 5, "Score Rush" (`objectives: [{ type: 'score', targetCount: 1000 }]`). The presentation layer (`Hud.tsx`/`WonOverlay.tsx`/`Home.tsx`) falls back to a fixed ★ glyph (`components/spriteAsset.ts`'s `SCORE_OBJECTIVE_SPRITE`) for a `'score'` objective's absent `targetMatchType`, rather than piping `undefined` through `getSpriteForMatchType` into the generic "?" placeholder this codebase already uses elsewhere as a genuine-bug signal. Verified live, see `docs/verification/score-objective/`. Still deferred (see `DEFERRED_COMPLEXITY.md`): `buildGeneratedLevelConfig` never produces a `'score'` objective (generated levels remain purely `'collect'`), and the point values/cascade-bonus rate are hand-picked, not playtested.

**A third objective type, `'clearance'`, is now built** (see `engine/DECISIONS.md`'s clearance-layers entry) — certain grid cells carry a hidden background layer (1 or 2, the level author's choice, mirroring blockers' `hitsToClear` convention) that decrements whenever the piece sitting on that cell is cleared by ANY effect — an ordinary match, a special sweep, a chain, a combo, a bomb blast — winning once every layered cell reaches 0. This is a genuinely new mechanic, not a blocker variant: a blocker's `hitsRemaining` lives on the immovable blocker piece itself, but a layer must stay pinned to the grid position, since `calculateCascades` refills a cleared cell with a freshly spawned piece of a different id — so it's tracked as a new `GameState.layerCells: Record<string, number>` (keyed `"row,col"`), separate from `Board`/`Piece` entirely (no new `PieceType`, `matrix.ts` untouched). The clear-pipeline reuse held cleanly, though: every existing clearing mechanism already funnels through `resolveCascades`'s per-pass loop or `resolveClearSet`'s shared tail, both of which already compute exactly the position set a layer decrement needs — `CascadeResolution` gained a `clearedPositions` field exposing that as raw positions (rather than `clearedByMatchType`'s per-matchType counts, since a layered cell isn't matchType-addressable), and `applyMove`'s `decrementLayers` does the actual decrement. A `'clearance'` objective's `targetCount` is never hand-authored — `createGameState` derives it by summing the level's own `layerCells`, so the two numbers can never drift out of sync. A layered cell is exclusively an ordinary/special matchable piece this session (never also a blocker or void — a confirmed, deliberate scope line, not an oversight). One hand-built level exercises it: App.tsx's `LEVEL_QUEUE` level 6, "Dusty Counter" (`objectives: [{ type: 'clearance' }]`, `layerCells: DUSTY_COUNTER_LAYERS` — six cells, four at 1 layer and two at 2 layers). The presentation layer needed genuinely new pieces: `components/Tile.tsx`'s `LayerOverlay` renders a calm, light dusting wash whose opacity scales with layers remaining (deliberately not `SpreadWarningOverlay`'s dark hazard wash — a layer is "something's underneath, reveal it," not a threat), and `Hud.tsx`/`WonOverlay.tsx`/`Home.tsx`/`levelProgress.ts` fall back to a fixed ▤ glyph (`components/spriteAsset.ts`'s `CLEARANCE_OBJECTIVE_SPRITE`) for a `'clearance'` objective's absent `targetMatchType`. Verified live, see `docs/verification/clearance-layers/`. Still deferred (see `DEFERRED_COMPLEXITY.md`): `buildGeneratedLevelConfig` never produces `layerCells`/a `'clearance'` objective (generated levels remain purely `'collect'`), a layered cell never coexists with a blocker or void, and the layer-count range/wash opacity are hand-picked, not playtested.

`livesLastRegenAt` can be spoofed by changing the device clock. This is a known, accepted tradeoff at this scale. Leave a comment noting it, do not spend time solving it.

**Three safety-net hardening fixes are now built**, from a real playability audit that flagged two of them as cheap-fix-for-catastrophic-consequence (see `engine/DECISIONS.md`'s "Three real-playability-audit safety fixes" entry and `docs/verification/safety-hardening/`): `engine/gameState.ts`'s `loadSave` now falls back to a fresh save (the same `applyLoadedSave(null)` path a dev reset already uses) on any corrupted or schema-invalid save blob instead of throwing — previously, a single malformed save would crash on every subsequent launch, permanently, with no recovery short of a full data wipe. `components/ErrorBoundary.tsx` is a new app-level `ErrorBoundary` (the one class component in an otherwise all-function-component tree, wired as the outermost wrapper in `App.tsx`) that turns any other uncaught crash into a calm "Something went wrong / Start Fresh" screen, with its actual logic extracted into a react-native-free `components/errorRecovery.ts` module (importing `'react-native'` fails to parse under this repo's ts-jest config, the same limitation `services/hapticsService.ts` already documents) so it's genuinely unit tested rather than only ever verified live. `engine/matrix.ts`'s `shuffle` — the one rescue mechanism a stuck board has — used to silently return its last random attempt even if still illegal after 100 tries; it now falls through to a deterministic, multiset-preserving repair tier (`repairShuffleViaSwaps` → `forceLegalMove`, both self-verifying against `checkMatches`/`checkSquares`/`hasLegalMoves` before ever returning) and only throws a descriptive error in a genuinely impossible case, caught by the new `ErrorBoundary` rather than corrupting state silently. `gameState.ts`'s own mid-play rescue call site wraps that call in a `try`/`catch` instead of letting a failed rescue crash the whole move — matching that call site's own existing "should read as silent and immediate, not an announced interruption" design intent, distinct from `generateLevel`'s level-creation-time call, which still throws loudly on a genuinely broken level config.

The **sound/haptics stub layer** is now built (see `engine/DECISIONS.md`'s sound/haptics stub-layer entry): `SaveData.soundEnabled`/`hapticsEnabled` are two independent optional booleans (both default `false`, matching the sound-off design constraint above and applying the same calm-by-default reasoning to haptics even though no equivalent user research exists for it specifically), toggled from a real Home card, and threaded into `Board.tsx` as props. `components/soundEffects.ts`'s `triggerPassEffects` — a plain function, not a hook, since it holds no render state — is called once per cascade pass from `Board.tsx`'s `animateCascade`/`runStep(i)` (the only place `applyMove`'s `steps` are available without widening Board's prop surface): the first pass (`i === 0`, the player's own swap) plays a "match" sound and fires a light haptic; later cascade passes play a "cascade" sound only, no haptic (repeated haptic pulses on a fast chain would read as a buzzy alarm, against the calm-not-frantic constraint below); the final pass additionally plays a "win" sound if the move won the level. **A real audio backend is now built too** (see `engine/DECISIONS.md`'s real-audio-backend entry): `services/expoAudioSoundService.ts`, backed by `expo-audio` (chosen over `expo-av`, which was deprecated in SDK 53 and is removed in SDK 55 — this project is on SDK 54), is the real `SoundService` `services/defaultSoundService.ts` now constructs, replacing the old `silentSoundService` singleton (`silentSoundService.ts` still exists and is still correctly used as the injectable fake in tests). One player per effect is created lazily and kept alive for the app's lifetime (a fixed pool of at most three, not the create-and-discard case `expo-audio`'s docs warn needs manual release). ## Design Constraints (from actual user research, not guessing)

- Sound defaults to off, with an easy one-tap mute. She specifically said Candy Crush's sound was distracting. This is now real, not just aspirational: Home's Sound/Haptics toggle card (`components/Home.tsx`) and `SaveData.soundEnabled`/`hapticsEnabled` (both default `false`) are built — see the Data Model Notes below and `engine/DECISIONS.md`'s sound/haptics stub-layer entry. The toggle now gates a real `expo-audio`-backed playback service (three synthesized match/cascade/win tones — see the real-audio-backend entry) and, on native, a real light haptic tap.
- Animation and pacing should read as calm and satisfying, not frantic. She plays to keep her mind occupied, not for an adrenaline hit. High-intensity combo particle effects are a bad fit for this specific player even if they're standard for the genre.
- Board renders close to edge to edge. A decorative frame around the grid eats tile size, and smaller tiles mean worse tap accuracy on a phone.

## Playtest Feedback Protocol

When real playtesting surfaces something that feels wrong, broken, or unsatisfying, follow this shape by default, no need to be told each time:


Investigate the actual cause before proposing a fix. A symptom's obvious explanation is often wrong. Check the real code, the real data, the real behavior, before writing anything.
Check whether this might share a root cause with something already built. Several real bugs in this project turned out to be two independently correct systems that were never checked against each other once they started interacting (the piece-type ramp and the objective gate, the recipe card art and the placeholder contract, the win overlay and the cascade step sequencing). When something feels wrong shortly after a related feature shipped, checking for this kind of overlap first is often faster than assuming a fresh, unrelated bug.
When a new feature builds on an old one, check whether the old feature's shortcut still holds. A duplicated decision, a copied piece of logic, a special case, any of these can quietly drift the moment a second implementation exists. If a fix requires testing a decision, and that decision is duplicated somewhere, collapse the duplication into one shared source before writing the test, don't test a stand-in copy.
Verify against real behavior, not just a passing test suite. A test proves the code does what the test checks. It doesn't prove a player can see it, feel it, or that it's wired into the actual running app. When animation, timing, or feel is involved, a live trace, a real screenshot sequence, or a frame-by-frame capture is the honest standard, not a simulated or mocked equivalent.
Don't default to genre convention without checking it against this game's actual principles. This game is built for one specific real player who plays to relax, not for excitement, and everything a player gets access to comes from playing well, never purchased. Genre-standard choices (stingy resource grants, aggressive urgency framing, unlimited ad-for-continue loops) are worth checking against these principles specifically, not assumed correct just because most other games do it that way.
Confirm before deciding on a genuine fork, don't guess silently. If a fix could reasonably go two different directions with real consequences either way, say so plainly and ask, rather than picking one and hoping it was the intended one.
Update DECISIONS.md, DEFERRED_COMPLEXITY.md, and CLAUDE.md's own scope notes as part of the same session, not as an afterthought. Documentation moves with the code that changed, always.


This protocol applies by default. A future prompt doesn't need to restate it, just describe the actual symptom and trust this standing expectation to shape the investigation.

## Git Workflow Discipline

This project works explicitly on main. No feature branches, no long-lived stashes, one continuous history everyone can trust.


Confirm the branch before starting any new work. A quick git status or git branch at the start of a session costs nothing and catches drift immediately, before it has a chance to compound into something tangled later.
Commit promptly once something is confirmed done and verified, don't stash it. A stash is invisible to anyone not already looking for it, and it's exactly what caused a real, genuinely tangled conflict once already, two separate features colliding on the same lines with no clean side to pick because the stash sat untouched while other work landed on top of it in the meantime. If work needs to pause mid-stream, commit it, even as a rough checkpoint, rather than stashing it and hoping to return before anything else changes underneath it.
Prefer small, frequent commits over large batches. A history where every single commit is independently correct and independently testable is worth more than it might seem, it makes any future conflict resolution dramatically easier, it survives an unpredictable disruption cleanly, and it makes tracing exactly when something changed straightforward rather than a guessing game.
If a conflict ever does arise, reconstruct the intended commits deliberately, don't resolve it as one merged blob. When two genuinely different pieces of work land on the same lines with no single correct side, stage each logical change back into its own clean commit by hand, verify each one in true isolation before it's created, the same discipline already proven out once, rather than combining everything into one commit and hoping the result is right.


This protocol applies by default going forward, the same way the Playtest Feedback Protocol already does, no need to restate it, just follow it.
