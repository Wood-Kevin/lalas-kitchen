# Deferred Complexity

One line per deferred idea, with a short reason it was deferred. Parked is fine. Forgotten is not.

- ~~Row clearers~~ — resolved: the striped piece (a run of exactly 4 spawns a `type: 'striped'` row/column clearer; matching it sweeps that line) is built — see engine/DECISIONS.md's striped-piece entry. Remaining special-piece scope still deferred, deliberately, so the special-piece tiers stay bounded and correct: (a) **sweep chaining** — a striped piece (or color bomb) caught in another's line sweep / detonation just clears, it doesn't fire its own effect; real genre chain reactions are a follow-up; (b) **L/T-shape combos** — overlapping straight runs are each handled independently (a 4-run's anchor cell wins if another run also wanted to clear it), with no combined/wrapped special for the intersection, and a color bomb only spawns from a *straight* 5, not an L/T intersection of two shorter runs meeting at a shared point; (c) **bomb + striped combos** — swapping a color bomb with a striped piece just clears every piece of that striped piece's matchType (the bomb's ordinary behavior), rather than the genre's "convert every piece of that color to striped and detonate them all" super-combo. (Blocker clearing was built earlier — see engine/DECISIONS.md's Phase 6 section.)
- ~~5+-match specials (color bomb)~~ — resolved: a run of **exactly 5** now spawns a `type: 'color_bomb'` piece; swapping it with any piece detonates every piece on the board sharing that piece's matchType (swapping two bombs clears the whole board). It activates on the swap itself, a genuinely different mechanism from the striped piece's match-triggered activation — see engine/DECISIONS.md's color-bomb entry, and `docs/verification/color-bomb/` for the live capture. A run of **6+** still clears normally with no larger special; the L/T-shape *trigger* for a color bomb (intersecting runs meeting at a shared point, rather than one straight run of length 5) is deferred alongside the L/T combos in the entry above.
- Hidden-piece-underneath blocker mechanic (a blocker conceals a real matchable piece; clearing the blocker layer reveals it, and the revealed piece has to be matched separately) — considered as an alternative to the adjacent-damage mechanic actually built for cling wraps (engine/DECISIONS.md, Phase 6), and rejected for now: it doubles the per-cell data model (a blocker cell would need its own state *and* a full hidden `Piece`) for a payoff — a second, meaningfully different blocker *feel* — that doesn't matter yet with only one blocker type in the skin. Worth revisiting once a second blocker type is real and "every blocker behaves identically" starts to feel thin.
- Recipe box meta layer UI — out of scope for V1 per CLAUDE.md; engine's end-of-level summary event should exist eventually but nothing listens to it yet.
- Cloud asset delivery / per-skin CDN loading — irrelevant until a second skin is real.
- ~~Multi-target objectives~~ — resolved: `GameState`/`LevelConfig`'s `objective` became `objectives`, an array (a single-objective level is length one, no special case); win requires every entry met (`engine/gameState.ts`'s `applyMove`); generated levels place a second objective past a difficulty threshold once the level's own piece-type pool has room for a distinct second target (`appPersistence.ts`'s `generatedObjectiveCount`). See `engine/DECISIONS.md` for the full reasoning. Score-threshold objectives (a numeric score, distinct from counting matched pieces) remain out of scope — nothing asked for that yet.
- App Store "distinct product" layout variation — matters only once skin number two ships.
- ~~What actually decrements `GameState.lives` during a level attempt~~ — resolved: losing a level (moves exhausted) now spends exactly one account-level life, applied in `App.tsx` (not mid-level in the engine) and gated at every level-start entry point (`canStartLevel`). The mid-level `pauseReason: 'lives'`/`grantBonusLife` mechanics Phase 4 built for the *other* candidate design (a mid-level lives resource with its own spend trigger) were removed outright rather than kept as unreachable insurance — see `engine/DECISIONS.md`'s Phase 7 section for the full reasoning on why that design was dropped instead of finished.
- Basic lives regen (`appPersistence.ts`'s `applyLivesRegen`) has no visible countdown anywhere — `OutOfLives.tsx` says "come back soon" but doesn't compute or show an actual "next life in N minutes" readout, even though every input needed to compute one (`lives`, `livesLastRegenAt`, `regenMinutes`) already exists. Skipped this session per the explicit "doesn't need real design polish, just needs to exist" scope for the out-of-lives state; a real countdown is a small, well-scoped follow-up once it's actually wanted.
- No screen shows the player's current lives count outside of an active level's HUD — Home and All Levels both now gate real gameplay on a live, reactive `lives` value (see `App.tsx`), which makes this a more meaningful gap than it was when this same idea was skipped in the entry below (back when the only available value was a persisted, not-live `SaveData.lives`). Still not built this session since it wasn't asked for, but the original reasoning for skipping it is now partly stale — worth a real look next time Home/All Levels are touched.
- ~~Per-step cascade animation~~ — resolved: `applyMove` now returns `steps: Board[]` (one settled board per cascade pass, last entry equals `state.board`), and `Board.tsx` walks them so each pass animates as its own beat. See engine/DECISIONS.md's cascade-steps entry.
- Literal edge-to-edge fill on all four sides of the board — with square tiles and an 8-row/6-col grid on a typical tall phone screen, tile size is bound by screen *width* (6 columns is the tighter constraint), so there's unavoidable vertical slack even at max tile size. `Board.tsx` now centers the grid in the full available height (see `components/NOTES.md`) rather than dumping all the slack below it, but it isn't literally edge-to-edge top and bottom unless tiles go non-square or the grid's row/col ratio is changed — neither of which was asked for.
- ~~Dashboard is a dead end once every level in `LEVEL_QUEUE` is complete~~ — resolved: past the hand-built queue, `buildLevelConfig` (`App.tsx`) falls through to `buildGeneratedLevelConfig` (`appPersistence.ts`) indefinitely instead of dead-ending, and any hand-built or previously-completed generated level can be replayed via `handlePlayLevel`. Superseded by the Home/All Levels rebuild below — the flat `Dashboard.tsx` this note originally described no longer exists; `resolveStartScreen` no longer "always resumes gameplay directly" either (see that entry).
- `LEVEL_QUEUE` in `App.tsx` is still a hardcoded 3-level array (seed/movesLimit/objective per level), not a real level-authoring pipeline — adding a 4th *curated* level or rebalancing an existing one still means hand-editing that array. No longer blocking on content running out, though: past index 3, levels are generator-driven and unbounded.
- Generated-level difficulty ramp (`generatedPieceTypeCount`/`generatedMovesLimit`/`generatedTargetCount` in `appPersistence.ts`) is a simple linear step-down with hand-picked floors (3 piece types, 18 moves) and a capped target (26) — not tuned against real playtesting, just enough to keep a long session from becoming either trivial or mechanically unwinnable. Revisit once there's actual play data past level ~10.
- `components/AllLevels.tsx`'s row list (formerly `Dashboard.tsx`) grows by one for every generated level the player finishes and never prunes — fine at this project's actual scale (one real player, no infinite-scroll concern), but not a paginated/virtualized list if that stops being true.
- Power-up tray UI — the empty `powerUpTray` strip (`Board.tsx` + `skinConfig.powerUpTray` + `config.json`) reserved space beneath the board for icons that were never designed or built, and was pulled entirely rather than left rendering nothing. Removing it lets `boardArea` reclaim that fixed height as (larger) centered slack above/below the board — the same pre-existing vertical-slack tradeoff already logged above, not a new issue. Real power-up UI is out of scope until actual power-up pieces/behaviors are designed (see the row-clearer/blocker entry above); rebuilding the tray at that point should size itself off real content instead of a placeholder height.
- The recipe book's "total" (Home's `{completed} of {total}` and All Levels' enumerable row list) is defined as `resolveVisibleLevelIndices` — the hand-built queue's 3 levels always, plus any generated level actually completed (`components/levelProgress.ts`). This was a deliberate call, not an oversight: the generator is unbounded (see `resolveNextLevelIndex`'s own note), so there's no real fixed content ceiling to lock against the way the design mockup's illustrative "24" implied — inventing a round number would have been exactly the kind of fabricated value this session was told not to hardcode. The real consequence: once a player is past the hand-built queue and current on generated levels, the progress dot row and "X of Y" always read as fully caught up (a completed generated level is only ever added to the list already-completed) — "locked" is only ever observable within the original 3 hand-built levels. Revisit if a real fixed level-count content plan ever exists.
- All Levels renders only two row states (completed/checkmark/tappable vs. locked/dimmed/not-tappable), per this session's written spec. The Claude Design interactive prototype (`skins/lalas-kitchen/design-reference/interactive-prototype.html`) actually shows a third "up next" state — the very next unplayed level gets a highlighted border and an "up next" pill instead of being locked like everything else not yet completed. Deliberately not built this session since the written requirements enumerated exactly two states; the immediate next level is still fully playable, just via Home's "Start cooking" rather than by tapping its row on All Levels. Worth a quick confirm if the three-state look is actually wanted.
- Home's hero banner fade is a dependency-free approximation of the design mockup's actual CSS gradient fade under the hero image. No gradient library (e.g. `expo-linear-gradient`) is installed, and none was added without being asked — `Home.tsx`'s hero fade is a stack of increasingly-opaque solid bands, visually close at the sizes involved but not pixel-identical to a real gradient. (`components/GinghamTrim.tsx`'s checkerboard trim was originally in this same entry as a 1D-stripe approximation — it's since been rebuilt as a genuine 2-row offset checkerboard, no longer an approximation of the mockup's pattern shape, still gradient-free.)
- The design mockups use two custom Google Fonts ('Baloo 2' for headings, 'Nunito Sans' for body) — neither is loaded in this app (no `expo-font`/`@expo-google-fonts/*` dependency exists yet). Home.tsx/AllLevels.tsx match the mockup's weight/size/color choices using the system default font instead. Loading the real fonts is a small, well-scoped follow-up (Expo has first-party packages for both) but wasn't part of this session's ask.
- The mockup's Home hero shows an optional lives/flame-count indicator overlaid on the top-right corner of the header image. Skipped — it isn't in this session's written Home screen requirements (only in the interactive prototype's decorative flourish). Originally also justified by "Home has no in-progress `GameState` to read a live lives count from" — that reasoning is now stale: `App.tsx` has carried a real, reactive account-level `lives` value since the life-spend session (see the entry above), so passing it into `Home` would just be a prop, not a new data source. Still not built since it wasn't asked for, but worth knowing the old blocker is gone if this is revisited.
