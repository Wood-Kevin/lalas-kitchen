One presentation nicety from that work is deferred: the **convert-to-striped flash** — combo 2 computes the settled clear directly (the union of the converted pieces' sweeps), so there's no intermediate frame that visibly shows every matching piece turn striped before clearing; the engine result is identical, only the mid-combo animation beat is missing. (Blocker clearing was built earlier — see engine/DECISIONS.md's Phase 6 section.)
- ~~5+-match specials (color bomb)~~ — resolved: a run of **exactly 5** now spawns a `type: 'color_bomb'` piece; swapping it with any piece detonates every piece on the board sharing that piece's matchType (swapping two bombs clears the whole board). It activates on the swap itself, a genuinely different mechanism from the striped piece's match-triggered activation — see engine/DECISIONS.md's color-bomb entry, and `docs/verification/color-bomb/` for the live capture. A run of **6+** still clears normally with no larger special; the L/T-shape *trigger* for a color bomb (intersecting runs meeting at a shared point, rather than one straight run of length 5) is deferred alongside the L/T combos in the entry above (the newly-built crossing-run trigger only ever spawns an area bomb from two exactly-3 arms; it never upgrades a 5-long arm's own color-bomb spawn).
- ~~2×2-square specials (area bomb)~~ — resolved: a **2×2 square** of four same-type pieces spawns a `type: 'area_bomb'` piece; **swapping it with any ordinary piece** fires the full **3×3** block around it. It's now a **colorless, swap-activated** special (drops its matchType), the same camp as the color bomb — this **reverses** its original passive/colored design (matching a colored bomb into a run), because the single universal `area_bomb.webp` sprite gave a player no way to see which color a passive bomb was; making it colorless makes the ambiguity moot. See engine/DECISIONS.md's area-bomb reversal entry and `docs/verification/area-bomb/active/` for the swap-triggered live capture. A 2×2 is a new match *shape* (no 3-in-a-row), so a `checkSquares` scan (`matrix.ts`) runs alongside `checkMatches` and every "is there a match?" gate consults both (`applyMove` snap-back, `resolveCascades` loop, `hasLegalMoves`, generator/`shuffle` guarantees) — otherwise a square-forming swap snaps back and latent squares spawn free bombs. ~~Still deferred: the L/T-shape trigger~~ — partially resolved: a genuine two-arm crossing-run (a horizontal and a vertical run, each EXACTLY length 3, sharing one cell — the classic L/T/plus) now spawns an area bomb at the shared cell too (`matrix.ts`'s `checkCrossShapes`, additive to `checkSquares`) — see engine/DECISIONS.md's crossing-run entry and `docs/verification/crossing-shape/`. Still deferred, narrower than before: a **square** overlapping a run (not two runs crossing) still stands down exactly as before; an intersection involving a 4- or 5-long arm also stands down entirely at that cell, leaving the existing striped/color-bomb run logic to fire unchanged — only an exact 3×3 crossing is in scope. (Blast chaining — an area bomb caught in another effect's clear firing its own 3×3 — is now **built**, see entry (a) above.) ~~A square with a live striped corner used to silently fail to form at all~~ — **resolved**: `checkSquares` now accepts a live striped corner (blocker/void still excluded), and it fires its own sweep instead of a new area bomb spawning — see engine/DECISIONS.md's "Squares: a live striped corner now counts" entry and `docs/verification/square-striped-corner/`.
- **area + special combos** — deferred, architect-confirmed. Now that the area bomb is swap-activated (entry above), swapping one directly into another special (a color bomb, a striped piece, or **another area bomb**) is two swap/match-activated specials meeting, with no defined combined effect yet. Rather than pick one, such a swap **snaps back with no move spent** (the same reject path a no-match ordinary swap uses), guarded in `applyMove` *before* the color-bomb branch (or `resolveColorBomb` would degenerately clear only the bomb cell, since the area bomb has no matchType) and mirrored in `hasLegalMoves` (area+special is not a legal move). Only `area + ordinary` fires the 3×3 blast. This keeps area-bomb combos bounded exactly like the still-deferred square-overlapping-run and 4/5-arm L/T-shape triggers above (the exact-3×3 crossing-run case is now built — see engine/DECISIONS.md's crossing-run entry). (Note this is distinct from chaining, now built: chaining is a special *caught in another's clear*, whereas this is two specials *swapped directly into each other* — an area bomb caught in a chain fires its 3×3, but swapping one directly into another special still snaps back.) A real combined effect (e.g. a bigger blast, or converting a type) is a follow-up.

- Dynamic denial-zone spread — the mechanic itself is built (a gated blocker zone that grows into an adjacent ordinary cell if left unaddressed, with a calm crack/dimming warning the move before — see engine/DECISIONS.md's Phase 8 and `docs/verification/denial-zone-spread/`). - Board shape / void cells — non-rectangular boards are now **built** (a `type: 'void'` sentinel piece makes cells non-playable; segmented per-column gravity treats a void as a fixed floor; matching/legal-moves/shuffle/generator/render are all void-aware; `LevelConfig.voidCells` carves the shape; the plus-shaped Level 4 "Cutting Board" showcases it — see engine/DECISIONS.md's board-shape entry and `docs/verification/board-shape/`). Deliberately bounded, deferrals logged rather than built: (a) **generator-driven shapes** — `generateLevel` only carves the `voidCells` it's *handed*; it never invents a shape, so every non-rectangular board is hand-authored today (the same prove-it-with-real-content-first sequencing the specials followed — generator integration is its own later step); ~~(b) voids × specials interaction~~ — **resolved**: a real playtest found that a sweep/blast/detonation reaching a void did **not**, in fact, simply skip it — five clear-set builders (the in-match striped sweep, chain expansion, the area-bomb blast, the color-bomb whole-board detonation, and the striped-combo clear-set helper) were written before voids existed and only ever excluded `'blocker'`, so on a shaped board they could clear straight through a void, which segmented gravity then permanently refilled as an ordinary gap. Fixed with one shared `isClearable` predicate at all five sites — see engine/DECISIONS.md's "Fixed: the ... claim above didn't hold" entry and `docs/verification/void-specials-clear-fix/`. Void × **denial-spread** interaction remains correctly excluded and un-exposed to this bug (the spread already only ever targets `'normal'` cells, never a void) — there's no special that *creates*, *removes*, or *moves* a void, and no shape that changes mid-level. (c) **the "board always full" invariant assumes reachable pockets** — segmented gravity refills each enclosed playable pocket from its own local top (a piece appears mid-board), which is correct and calm but means a shape author can't rely on refills *flowing* between pockets separated by a void; a diagonal/flow gravity model was explicitly rejected as overkill (see the DECISIONS entry).
